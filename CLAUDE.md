# CLAUDE.md

Этот файл предоставляет инструкции для Claude Code (claude.ai/code) при работе с данным репозиторием.

## Описание проекта

Простой VPN-туннель на C++17 для Linux. Два бинарника: `vpn-server` и `vpn-client`, оба требуют запуска от root. Трафик шифруется по UDP с использованием XChaCha20-Poly1305 (libsodium).

## Сборка

```bash
# Установка зависимостей (Ubuntu/Debian)
apt install -y build-essential cmake libboost-program-options-dev libsodium-dev libfmt-dev iproute2 iptables openssl

# Сборка
mkdir build && cd build
cmake -DCMAKE_BUILD_TYPE=Release ..
cmake --build .

# Бинарники
build/src/server/vpn-server
build/src/client/vpn-client
```

В CLion используется существующая директория `cmake-build-debug/`.

Автоматических тестов в проекте нет.

## Стиль кода

Форматирование через clang-format (Google-based стиль). Основные параметры: отступ 4 пробела, лимит строки 150 символов, `PointerAlignment: Left`. Перед коммитом запускать `clang-format -i` на изменённых файлах.

## Коммиты

Сообщения коммитов пишутся **на русском языке** первая строка кратко, вторая строка пустая, третья и последующие описание более подробно, чтобы было понятно что сделано.

Каждое сообщение должно начинаться с префикса типа изменения:

| Префикс | Когда использовать |
|---|---|
| `feat:` | новая функциональность |
| `fix:` | исправление бага |
| `docs:` | изменения документации |
| `style:` | форматирование, отступы, clang-format (без изменения логики) |
| `refactor:` | рефакторинг без изменения поведения |
| `test:` | добавление или изменение тестов |
| `build:` | изменения системы сборки (CMake, зависимости) |
| `ci:` | изменения CI/CD |
| `chore:` | прочие служебные изменения |

Примеры:

## Архитектура

### Структура исходников

```
src/
  share/       # Общий код, компилируется в оба бинарника
    crypto.cpp/hpp   # Класс Crypto — шифрование/расшифровка XChaCha20-Poly1305
    device.cpp/hpp   # Класс TunDevice — открытие /dev/net/tun
    os.cpp/hpp       # Класс Os — настройка маршрутов, тюнинг UDP, syslog
  server/
    server.cpp       # Класс VPNServer + main()
  client/
    client.cpp       # Класс VPNClient + main()
```

### Модель потоков

**Сервер** запускает три потока и главный цикл статистики:
- `tunToUdpLoop` — читает из TUN, шифрует, отправляет нужному клиенту по UDP (поиск в `m_clients` по destination IP)
- `udpToTunLoop` — принимает UDP, расшифровывает, пишет в TUN; регистрирует/обновляет клиентов в `m_clients` (ключ — TUN IP в network byte order, `uint32_t`)
- `cleanupLoop` — удаляет клиентов с `last_seen` старше 60 секунд

**Клиент** запускает два потока и главный цикл keep-alive/статистики:
- `tunToUdpLoop` — читает TUN, шифрует, отправляет на UDP-endpoint сервера; пропускает пакеты с назначением `192.168.200.{1,2,3}` (подсеть VPN)
- `udpToTunLoop` — принимает UDP, расшифровывает, пишет в TUN
- Главный поток каждые 10 с отправляет нулевой keep-alive, чтобы сервер знал endpoint клиента

### Формат зашифрованного пакета

```
[POMA: 4 байта magic][nonce: 24 байта][ciphertext + auth tag]
```

Класс `Crypto` оборачивает `crypto_aead_xchacha20poly1305_ietf_*` из libsodium. Общий ключ передаётся как base64-строка и декодируется через `sodium_base642bin`. `Crypto::overhead()` возвращает 44 байта (4 + 24 + 16) — это учитывается при расчёте MTU (рекомендуется 1380).

### Настройка сети

- **Сервер**: `Os::setupServerRoutes` включает `ip_forward` и добавляет правила iptables MASQUERADE + FORWARD через определённый интерфейс по умолчанию.
- **Клиент**: сохраняет исходный шлюз до вызова `Os::setupClientRoutes`, который заменяет маршрут по умолчанию на `tun0`. UDP-сокет привязывается к физическому интерфейсу через `SO_BINDTODEVICE`, чтобы VPN-трафик сам не шёл через туннель.
- Подсеть клиентов захардкожена как `192.168.200.0/24`; сервер проверяет это в `udpToTunLoop` через маску подсети.

### Логирование

Оба бинарника пишут в syslog. Просмотр:
```bash
journalctl -f -t vpn-server
journalctl -f -t vpn-client
```

### Systemd-юниты

Шаблоны сервисных файлов находятся в `src/server/systemd/` и `src/client/systemd/`. Перед деплоем нужно прописать абсолютные пути в `ExecStart`.

## Известные ограничения

- Нет защиты от replay-атак (нет счётчиков пакетов и anti-replay window)
- Нет handshake; ключ передаётся через аргумент `--key` (виден в `ps`)
- Нет ротации ключей и разделения ключей по направлениям
